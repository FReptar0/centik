Define 6 enums from DFR.md:
- TransactionType: INCOME, EXPENSE
- Frequency: QUINCENAL, MENSUAL, SEMANAL, VARIABLE
- DebtType: CREDIT_CARD, PERSONAL_LOAN, AUTO_LOAN, OTHER

- CategoryType: EXPENSE, INCOME, BOTH
- AssetCategory: PPR, INVESTMENT, SAVINGS, CRYPTO, OTHER

Define 7 MVP models exactly matching DFR.md sections 2.1-2.7:

1. **IncomeSource** (DFR 2.1): id (cuid), name (String), defaultAmount (BigInt), frequency (Frequency enum), type (String — store as "EMPLOYMENT", "FREELANCE", "OTHER" — NOT an enum since DFR.md does not define a named enum for this), isActive (Boolean, default true), createdAt, updatedAt. Add `@unique` on name for seed upsert idempotency. Has relation: transactions Transaction[].

2. **Transaction** (DFR 2.2): id (cuid), type (TransactionType), amount (BigInt), description (String?), categoryId (String), incomeSourceId (String?), date (DateTime @db.Date), paymentMethod (PaymentMethod?), periodId (String), notes (String?), createdAt, updatedAt. Relations: category Category, incomeSource IncomeSource?, period Period. Indexes: @@index([periodId, date]), @@index([categoryId]).

3. **Category** (DFR 2.3): id (cuid), name (String @unique), icon (String), color (String), type (CategoryType), isDefault (Boolean @default(false)), isActive (Boolean @default(true)), sortOrder (Int @default(0)). Has relations: transactions Transaction[], budgets Budget[].

4. **Debt** (DFR 2.4): id (cuid), name (String @unique), type (DebtType), currentBalance (BigInt), creditLimit (BigInt?), annualRate (Int — basis points), minimumPayment (BigInt?), monthlyPayment (BigInt?), originalAmount (BigInt?), remainingMonths (Int?), cutOffDay (Int?), paymentDueDay (Int?), isActive (Boolean @default(true)), createdAt, updatedAt. Add @unique on name for seed upsert.

5. **Budget** (DFR 2.5): id (cuid), categoryId (String), quincenalAmount (BigInt), periodId (String), createdAt, updatedAt. Relations: category Category, period Period. @@unique([periodId, categoryId]).

6. **Period** (DFR 2.6): id (cuid), month (Int), year (Int), startDate (DateTime @db.Date), endDate (DateTime @db.Date), isClosed (Boolean @default(false)), closedAt (DateTime?), createdAt (DateTime @default(now())). Relations: transactions Transaction[], budgets Budget[], monthlySummary MonthlySummary?. @@unique([month, year]). NOTE: Remove the updatedAt field that exists in the stub — DFR.md does not specify it for Period.

7. **MonthlySummary** (DFR 2.7): id (cuid), periodId (String @unique), totalIncome (BigInt), totalExpenses (BigInt), totalSavings (BigInt), savingsRate (Int — basis points), debtAtClose (BigInt), debtPayments (BigInt), notes (String?), createdAt (DateTime @default(now())). Relation: period Period.

Define 3 v2 models from DFR.md sections 2.8-2.10:

8. **ValueUnit** (DFR 2.8): id (cuid), code (String @unique), name (String), precision (Int), symbol (String?), providerUrl (String?), providerPath (String?), providerHeaders (Json?), refreshInterval (Int @default(0)), isActive (Boolean @default(true)), createdAt, updatedAt. Relations: rates UnitRate[], assets Asset[].

9. **UnitRate** (DFR 2.9): id (cuid), unitId (String), date (DateTime @db.Date), rateToMxnCents (BigInt), rateRaw (String), source (String), createdAt (DateTime @default(now())). Relation: unit ValueUnit. @@unique([unitId, date]). Add @@index([unitId]) for lookup queries.

10. **Asset** (DFR 2.10): id (cuid), name (String), unitId (String), amount (BigInt), category (AssetCategory), institution (String?), notes (String?), isActive (Boolean @default(true)), createdAt, updatedAt. Relation: unit ValueUnit. @@index([unitId]).

CRITICAL DETAILS:
- All monetary fields are BigInt (centavos). Interest rates are Int (basis points).
- All IDs use @id @default(cuid()).
- DateTime fields for dates that are date-only (no time) use @db.Date.
- createdAt uses @default(now()), updatedAt uses @updatedAt.
- The IncomeSource type field is a plain String (not enum) because DFR.md lists values ("EMPLOYMENT", "FREELANCE", "OTHER") but does not name a formal enum. Store as String to match the spec.
- Relations must have proper FK fields and @relation decorators.
- Foreign key cascading: Use default behavior (restrict on delete) for all FK relationships. Assets and UnitRates should cascade delete when their parent ValueUnit is deleted — add onDelete: Cascade to Asset.unit and UnitRate.unit relations.
