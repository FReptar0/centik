<objective>
Wire the two-step login flow (TOTP-03) with rate limiting (TOTP-05) by:

1. **Extending `authorizeUser`** to accept `totpCode` and `challenge` credentials. When the user has `totpEnabled = true`, `authorize` requires both the original password AND a valid challenge-bound totpCode (or backup code). When 2FA is disabled, existing single-factor behavior is preserved.

2. **Splitting `loginAction`** into a step-1 handler that bcrypt-compares the password, rate-limits by `email:ip`, and — for 2FA users — returns `{ requiresTotp: true, challenge, callbackUrl }` INSTEAD of issuing a session. For 1FA users the existing `signIn` + NEXT_REDIRECT path is preserved.

3. **Adding `verifyTotpAction`** that validates the signed challenge, rate-limits by `userId:ip`, auto-detects 6-digit TOTP vs 8-hex backup code, and calls `signIn('credentials', { email, challenge, totpCode: code })` to let `authorizeUser` finalize the session.

4. **Modifying `LoginForm`** to render `<TotpStep>` in place of the password form when the step-1 result returns `requiresTotp` + a challenge string. Same `/login` route; no new page (D-17).

5. **Creating `TotpStep`** — a client component with FloatingInput for the code, a "Usar codigo de respaldo" toggle that swaps placeholder + inputMode, and hidden inputs (`challenge`, `callbackUrl`, `email`) that feed into `verifyTotpAction` via `useActionState`.

6. **Extending tests** — `src/auth.test.ts` adds 2FA-branch cases; `src/actions/auth.test.ts` adds `loginAction` requiresTotp + rate-limit + `verifyTotpAction` end-to-end unit cases.

All error paths collapse to one generic Spanish string (`'Credenciales invalidas'` in step 1, `'Codigo invalido'` in step 2). NEXT_REDIRECT discipline: catch `AuthError` only; re-throw everything else (Pitfall 1 — Phase 26/28 re-learned).

**Purpose:** This is the authentication flow itself — every 2FA-enabled user's login passes through these changes. Without it, TOTP-03 is not delivered.

**Output:** Extended `authorize` + split Server Actions + 2-step login UI + unit tests. Integration tests (full login happy path with real DB) are deferred to Plan 29-05; this plan proves behavior at the unit/mock layer.
</objective>

             id: true,
             email: true,
             hashedPassword: true,
             isApproved: true,
             totpEnabled: true,
           },
         })

           return { error: 'Credenciales invalidas' }
         }

           return { error: 'Credenciales invalidas' }

         const callbackUrl = (formData.get('callbackUrl') as string) || '/'

         // 1FA user — existing flow: signIn throws NEXT_REDIRECT on success
         if (!user.totpEnabled) {
           try {
             await signIn('credentials', {
               email: parsed.data.email,
               password: parsed.data.password,
               redirectTo: callbackUrl,
             })
             return {}
           } catch (error) {
             if (error instanceof AuthError) return { error: 'Credenciales invalidas' }
             throw error // CRITICAL: re-throw NEXT_REDIRECT (Pitfall 1)
           }
         }

         // 2FA user — return signed challenge; client swaps to TotpStep
         return {
           requiresTotp: true,
           challenge: signChallenge(user.id, user.email),
           callbackUrl,
         }
       }
