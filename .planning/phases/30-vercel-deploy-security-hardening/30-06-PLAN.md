    ## 1. Deploy Runbook (First-Time Production Setup)

    ### 1.1 Provision Vercel Marketplace Prisma Postgres

    1. Log into https://vercel.com
    2. Select the Centik project (or create it via "Add New..." → "Project" → import from GitHub)
    3. Navigate to **Storage** → **Connect Database** → **Prisma Postgres**
    4. Follow the Marketplace flow to create a new Prisma Postgres instance. Select the closest region (Mexico operator → `us-east-1` / `iad1` for lowest latency with current Vercel default).
    5. After provisioning, Vercel auto-injects `DATABASE_URL` (the **pooled** connection) into the project's Environment Variables.
    6. In the Prisma Console (the URL Vercel shows after provisioning), find the **direct connection string** (hostname `db.prisma.io`, not `pooled.db.prisma.io`). Copy it.

    ### 1.2 Set Environment Variables in Vercel Project Settings

    Navigate to **Project** → **Settings** → **Environment Variables**. Add the following with scope **Production** (and **Preview** for `DIRECT_URL` only if you plan to run previews against the same Prisma Postgres — usually NOT recommended):

    | Variable | Source | Value |
    |----------|--------|-------|
    | `DATABASE_URL` | Auto-injected | (leave as-is; managed by Prisma Postgres integration) |
    | `DIRECT_URL` | Manual (copy from Prisma Console) | postgres://...@db.prisma.io:5432/... |
    | `AUTH_SECRET` | Generate locally | `openssl rand -hex 32` (minimum 32 chars) |
    | `AUTH_TOTP_ENCRYPTION_KEY` | Generate locally | `openssl rand -hex 32` (exactly 64 hex chars) |
    | `ADMIN_EMAIL` | Your email | fmemije00@gmail.com |
    | `ADMIN_PASSWORD` | Your secure password | minimum 12 characters |
    | `UPSTASH_REDIS_REST_URL` | Upstash Console | https://...upstash.io |
    | `UPSTASH_REDIS_REST_TOKEN` | Upstash Console | your token |

    Do NOT set `RATE_LIMIT_DISABLED` in production — `src/lib/env.ts` superRefine rejects `RATE_LIMIT_DISABLED=true` when `NODE_ENV=production` (D-20). If you accidentally set it, the build will fail with a clear Zod message.

    Do NOT set `DEBUG` in production — `src/lib/env.ts` (per D-30) logs a warning if detected. Optional; ignore unless debugging.

    ### 1.3 Override the Build Command

    Navigate to **Project** → **Settings** → **Build & Development Settings** → **Build Command** → toggle **Override** → set:
    ```
    npx prisma migrate deploy && next build
    ```
    `prisma migrate deploy` is idempotent — if there are no pending migrations it exits 0 immediately. It runs against `DIRECT_URL` via the new `prisma.config.ts` datasource (Plan 30-01). The `postinstall` script (`prisma generate`) already runs on every `npm install`; no additional change needed.

    Install Command and Output Directory: leave at defaults (`npm install` and Next.js default).

    ### 1.4 Trigger First Production Deploy

    Push to `main` branch (or trigger manual redeploy). Build must succeed — if Zod env validation fails, the build fails with a clear error in the Vercel log. Fix the missing env var and re-deploy.

    ### 1.5 Run the Production Admin Seed (One-Time)

    From your local machine:
    ```
    npx vercel link              # if not already linked
    npx vercel env pull .env.production
    ADMIN_EMAIL=$(grep ADMIN_EMAIL .env.production | cut -d= -f2) \
    ADMIN_PASSWORD=$(grep ADMIN_PASSWORD .env.production | cut -d= -f2) \
    DATABASE_URL=$(grep DATABASE_URL .env.production | cut -d= -f2) \
    npm run db:seed:prod
    ```
    Expected output on first run: `Admin user <email> created.`

    Re-run the same command — expected output: `Admin user <email> already exists -- flags verified (password NOT rotated).` (D-10)

    DO NOT re-run this command in a CI job or wire it into the build command. It is a one-shot (D-11).

    ## 2. Smoke Checklist (11 Items — D-27)

    Run these 11 items IN ORDER against the newly-deployed production URL (`https://<app>.vercel.app`). Every item has an expected outcome; mark pass/fail. If any item fails, file a follow-up issue before promoting the deploy.

    ### 2.1 Unauthenticated Redirect
    1. Visit `https://<app>.vercel.app` in a fresh browser (or incognito window).
       **Expected:** Redirected to `/login`. Proxy.ts is operational.

    ### 2.2 Admin Login
    2. Enter `ADMIN_EMAIL` + `ADMIN_PASSWORD` on `/login`. Submit.
       **Expected:** Lands on the dashboard (`/`).

    ### 2.3 Create a Transaction
    3. Click the FAB (bottom-right floating action button). Fill in a simple EXPENSE transaction (e.g. 100.00 MXN, category "Comida" — if no category exists yet, create one first via `/configuracion`).
       **Expected:** Transaction appears in Movimientos list. Dashboard KPIs update.

    ### 2.4 2FA Setup Visible
    4. Navigate to `/configuracion`. Scroll to the **Seguridad** section.
       **Expected:** "Activar 2FA" button is visible.

    ### 2.5 2FA Enable + QR + Backup Codes
    5. Click "Activar 2FA". Step 1 displays a QR code + manual secret. Scan with a real authenticator app (Google Authenticator, 1Password, Authy, etc.). Enter the 6-digit code from the app. Step 2 displays 10 backup codes — use the Download button to save them as a `.txt` file (the browser triggers a blob download).
       **Expected:** 2FA is enabled. Backup codes file downloads. Seguridad section now shows "Regenerar codigos de respaldo" and "Desactivar 2FA".

    ### 2.6 Logout
    6. Click the user menu → Logout.
       **Expected:** Redirected to `/login`.

    ### 2.7 Two-Step TOTP Login
    7. Enter `ADMIN_EMAIL` + `ADMIN_PASSWORD` again. After password step, you land on the TOTP step. Enter the current 6-digit code from your authenticator.
       **Expected:** Lands on dashboard.

    ### 2.8 Backup-Code Login
    8. Logout. Login with email + password. On the TOTP step, enter one of your saved backup codes (format `ab12-cd34` or `ab12cd34` — both accepted).
       **Expected:** Lands on dashboard. The backup code is now consumed (single-use). Re-attempting it would fail.

    ### 2.9 Security Headers Inspection
    9. From a terminal:
       ```
       curl -I https://<app>.vercel.app/login
       ```
       **Expected headers:**
       - `content-security-policy: default-src 'self'; script-src 'self' 'nonce-...' 'strict-dynamic'; ...`
       - `strict-transport-security: max-age=63072000; includeSubDomains; preload`
       - `x-frame-options: DENY`
       - `x-content-type-options: nosniff`
       - `referrer-policy: strict-origin-when-cross-origin`
       - `permissions-policy: camera=(), microphone=(), geolocation=(), browsing-topics=()`

       Contrast: `curl -I http://localhost:3000/login` should NOT show `strict-transport-security` (dev mode).

       Also verify in a browser at `https://<app>.vercel.app/` → DevTools Console → confirm **zero CSP violations** on the dashboard page.

    ### 2.10 Rate-Limit Trigger
    10. On `/login`, submit wrong password 6 times rapidly (within ~30 seconds) with the same email. The 6th attempt must show the generic form error message ("Credenciales invalidas" or equivalent). The sliding-window rate limiter (5 req/60s) has fired via real Upstash.
        **Expected:** 6th submit shows the generic error before hitting the DB. Network tab shows a non-successful response.

    ### 2.11 Invite User B + Cross-User Isolation Verification
    11. Logged in as admin, navigate to `/configuracion` → Invitaciones section. Click "Generar invitacion" — copy the invite URL. Paste into a fresh browser (different profile or incognito). Register a new user (email + password). Log in as User B. Visit `/movimientos`, `/presupuesto`, `/historial`.
        **Expected:** All three pages show zero rows belonging to the admin. User B sees only their own (empty) state.

    ## 3. HSTS Preload Submission (After 1 Week of Stable HTTPS)

    After the production deploy has been live for at least one week with stable HTTPS (no cert regressions, no redirect loops):

    1. Visit https://hstspreload.org
    2. Enter the production domain (e.g. `centik.vercel.app`, or your custom domain if configured later)
    3. The site verifies our HSTS header meets preload requirements:
       - `max-age >= 31536000` (1 year) — we emit `63072000` (2 years) ✓
       - `includeSubDomains` directive ✓
       - `preload` directive ✓
       - Valid cert on root domain
       - HTTP→HTTPS redirect on root domain
       - All subdomains serve HTTPS
    4. Submit for inclusion. Approval typically takes ~12 weeks to propagate to Chrome/Firefox/Safari/Edge preload lists.
    5. Once submitted, any cert downgrade or domain abandonment is hard to reverse — removal from preload lists takes months. Only submit when the domain is committed long-term.

    Optional: skip preload submission if Centik is a personal deployment without a long-term custom domain — HSTS without preload still protects every returning visitor.
