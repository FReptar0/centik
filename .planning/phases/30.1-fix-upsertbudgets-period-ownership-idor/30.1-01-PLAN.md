<objective>
Close the `upsertBudgets` partial-IDOR discovered by Phase 30 Plan 05. Add Phase 27's canonical `findFirst({ where: { id, userId } })` ownership-guard pattern to both the `periodId` argument AND every submitted `categoryId` inside `entries[]` so no cross-user Budget row can be created — returns ambiguous `_form` error consistent with the rest of the auth/IDOR surface.
</objective>

<deliverable>
One Prisma ownership-guard chain at the top of `upsertBudgetsAction` + inline Zod-error-shape error returns, plus unit tests for the two new rejection paths and one integration test proving the attack is blocked end-to-end.
</deliverable>

<context_summary>
- The bug: `upsertBudgets` in `src/app/(app)/presupuesto/actions.ts` runs `prisma.budget.findFirst({ where: { periodId, categoryId, userId } })` — when the submitted `periodId` belongs to another user, this returns null (correctly), but the action falls through to the `create` branch and inserts a new Budget row linked to the victim's period + category but owned by the attacker. No victim data is leaked or overwritten, but stale cross-user rows accumulate — which violates the per-user discipline that Phases 27/29 established.
- The fix: enforce ownership BEFORE any write by running `findFirst({ where: { id, userId } })` on both the `Period` record (once) and each submitted `Category` record.
- Analog pattern: Phase 27 Plan 02 (`updateTransaction` IDOR fix — `findFirst` pre-check before update), Phase 28 `requireAdmin` fresh-DB-lookup pattern. Both proved this shape in production.
- The fix is a single-file production change + 2 test files. No schema change, no new deps, no migration.
</context_summary>

    ### A.1 — Period ownership guard

    ```typescript

      where: { id: periodId, userId },
    ```

    Rationale: single DB round-trip confirms the authenticated user owns the submitted periodId. `select: { id: true }` minimizes payload.

    ### A.2 — Category ownership guard (batch)

    ```typescript
    const submittedCategoryIds = parsed.data.entries.map((e) => e.categoryId)
    const ownedCategories = await prisma.category.findMany({
      where: { id: { in: submittedCategoryIds }, userId },
      select: { id: true },
    })
    if (ownedCategories.length !== submittedCategoryIds.length) {
      return { error: { _form: ['Categoria no encontrada'] } }
    }
    ```

    Rationale: single batched DB round-trip covers all submitted categoryIds; returning a count-mismatch implies at least one categoryId is not owned by the user — exact mismatch identity is withheld per the ambiguous-error rule (Phases 26/28/29).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Authenticated user session -> Server Action | `upsertBudgets` receives `periodId` + `entries[].categoryId` from client form submission; must verify ownership before any write |
| Server Action -> Prisma write | Per Phase 27 D-IDOR: `findFirst({ where: { id, userId } })` pre-check is the canonical guard; Plan 30.1 applies it uniformly |

## STRIDE Register

| ID | STRIDE | Surface | Severity | Mitigation |
|----|--------|---------|----------|------------|
| T-30.1-IDOR-001 | Spoofing / Tampering | `upsertBudgets.periodId` parameter | HIGH (IDOR, data pollution) | Task 1 Part A.1 — `prisma.period.findFirst({ where: { id: periodId, userId }})` ownership guard |
| T-30.1-IDOR-002 | Spoofing / Tampering | `upsertBudgets.entries[].categoryId` parameter | HIGH (IDOR, data pollution) | Task 1 Part A.2 — batched `prisma.category.findMany({ where: { id: { in: ... }, userId }})` ownership guard |
| T-30.1-INFO-001 | Information Disclosure | Error messages from rejected IDOR attempts | LOW | Ambiguous Spanish error strings (`Periodo no encontrado` / `Categoria no encontrada`) per Phases 26/28/29 oracle-resistance rule |

## Risk Register

| Risk | Impact | Mitigation |
|------|--------|------------|
| Fix breaks same-user happy path | Regression | Task 1 Part B.3 regression guard + Task 2 Part B full quality gate including existing isolation tests |
| Fix introduces new error-shape that UI doesn't handle | UX regression | Error shape matches existing `{ error: { _form: [...] } }` pattern already consumed by the presupuesto client wrapper — zero UI change needed |
| Batched category lookup exposes timing oracle | Very low | findMany is O(1) in Postgres with userId + id in the where clause; unlikely to leak meaningful timing |
</threat_model>
