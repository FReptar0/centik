**src/lib/period.ts** (3 functions):

1. Private `findOrCreatePeriod(month, year)` -> `findOrCreatePeriod(month, year, userId: string)`
   - Add `userId` to `prisma.period.findUnique()` -- NOTE: findUnique uses the composite key `month_year`, which does NOT include userId. Instead, after finding, verify the period belongs to the user OR create with userId.
   - IMPORTANT: Period is shared across users in the current architecture. For the expand-contract pattern, add userId to the `create` data only. The findUnique uses the composite key `{ month_year: { month, year } }`. For per-user isolation (Phase 27), we will need to change Period lookup to include userId in the composite key. For NOW in Phase 25: just add userId to create data, and pass userId through the function signatures.
   - Update: `prisma.period.create({ data: { month, year, startDate, endDate, isClosed: false, userId } })`
   - Update findUnique to findFirst with userId: `prisma.period.findFirst({ where: { month, year, userId } })`

2. `getCurrentPeriod()` -> `getCurrentPeriod(userId: string)`
   - Pass `userId` to `findOrCreatePeriod(month, year, userId)`

3. `getPeriodForDate(dateStr)` -> `getPeriodForDate(dateStr: string, userId: string)`
   - Pass `userId` to `findOrCreatePeriod(month, year, userId)`

**src/lib/history.ts** (3 functions):

   - Add `userId` to `prisma.monthlySummary.findMany({ where: { period: { year } } })` -> `where: { period: { year }, userId }` OR `where: { period: { year, userId } }` depending on whether MonthlySummary has userId or needs to filter through period relation. Since MonthlySummary now has its own userId, use: `where: { userId, period: { year } }`

   - Add `userId` to `prisma.period.findMany()` -> filter periods by userId

**src/lib/budget.ts** (2 functions):

   - Add `userId` to budget findMany and transaction groupBy

2. `copyBudgetsFromPreviousPeriod(periodId, month, year)` -> `copyBudgetsFromPreviousPeriod(periodId: string, month: number, year: number, userId: string)`
   - Add `userId` to period findUnique (use findFirst with userId), budget findMany, and budget createMany

NOTE on Period uniqueness: The current schema has `@@unique([month, year])` on Period. With multi-user, each user can have their own period for the same month/year. This means the `@@unique([month, year])` constraint should become `@@unique([month, year, userId])`. However, since userId is still optional (String?) in this step, we CANNOT change the unique constraint yet. Instead, use `prisma.period.findFirst({ where: { month, year, userId } })` instead of `findUnique({ where: { month_year: { month, year } } })`.

The unique constraint update will happen in Task 3 when userId becomes required.
