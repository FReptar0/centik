- Replace the loan progress bar div block with:
  ```tsx
  <BatteryBar value={percentPaid} variant="compact" thresholds={{ warning: 25, danger: 50 }} />
  ```
  Note: loan progress is inverted conceptually (higher = more paid = better), but the existing code already handles this with the `progressColor` logic. Use thresholds that match the existing color bands: red <25%, orange 25-50%, green >50%.
  Wait -- existing logic uses: `percentPaid > 50 ? 'positive' : percentPaid >= 25 ? 'warning' : 'negative'`. BatteryBar colors segments by end position vs thresholds. For loans, higher values should be green. BatteryBar's default behavior is green < warning, orange warning-danger, red > danger. So for loans where high = good, we need the BatteryBar to show green for low fill and green for high fill... This doesn't work with BatteryBar's traffic-light model which is designed for "budget spent" (low = good, high = bad).

  **Resolution:** For loan payoff, the traffic-light model doesn't directly apply (higher paid = better). Keep the existing progressColor logic for the text label. For the BatteryBar, since BatteryBar always shows traffic-light (low=green, high=red), use it as-is with `thresholds={{ warning: 80, danger: 100 }}` (same as budget) -- this means a loan that's 90% paid would show orange/red segments, which is incorrect for a "progress toward payoff" bar.

  **Better approach:** Since BatteryBar's color model assumes "low is good" and loan payoff assumes "high is good", render BatteryBar with all-accent coloring by setting impossibly high thresholds: `thresholds={{ warning: 101, danger: 102 }}`. This makes ALL filled segments accent (chartreuse) regardless of fill level, which is correct for a payoff progress indicator where any progress is good. The text above still uses the existing progressColor for the percentage label.
- Keep the percentage text that uses SEMANTIC_COLORS[progressColor]
