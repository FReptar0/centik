  <action>
    Perform a comprehensive WCAG 2.1 AA accessibility audit across ALL components and pages. Read every component file, checking each against the accessibility requirements below. Fix any violations found.

    **1. Contrast Ratios (STYLE_GUIDE.md > Paleta de Color > Ratios de contraste):**
    - Verify `--color-text-primary` (#E8E8E8) on `--color-bg` (#000000) = 17.4:1 (passes AAA) -- used for all primary text
    - Verify `--color-accent` (#CCFF00) on `--color-bg` (#000000) = 14.4:1 (passes AAA) -- used for interactive elements
    - Verify `--color-text-secondary` (#999999) on `--color-bg` (#000000) = 5.8:1 (passes AA) -- used for labels and descriptions
    - Verify `--color-text-tertiary` (#666666) on `--color-bg` (#000000) = 3.7:1 -- this ONLY passes for large text (18px+) or decorative text. Audit ALL uses of `--color-text-tertiary` / `text-text-tertiary` to ensure they are NEVER used for essential content at small sizes. The "$" prefix in MoneyAmount is acceptable (decorative).
    - Check `--color-accent` (#CCFF00) text on `--color-surface-elevated` (#141414) -- must meet 4.5:1 for normal text
    - Check `--color-text-secondary` (#999999) on `--color-surface-elevated` (#141414) -- must meet 4.5:1 for normal text
    - If any contrast ratio fails, either change the text color or increase the background contrast

    **2. Focus Rings (STYLE_GUIDE.md > Elevacion > Focus Rings):**
    - Global `:focus-visible` rule must exist in globals.css: `outline: 2px solid var(--color-accent); outline-offset: 2px`
    - Verify EVERY interactive element (buttons, links, inputs, toggles, numpad keys) receives visible focus ring when Tab-navigated
    - No focus trap in modals (Tab should cycle through modal content, Escape should close)
    - Focus order must be logical: top-to-bottom, left-to-right within each page
    - Check that no component overrides `:focus-visible` to remove the outline without providing an alternative
    - Specifically check: Numpad keys, TogglePills options, BatteryBar (not focusable -- correct), FAB button, modal close button

    **3. ARIA Attributes on New Components:**
    - BatteryBar: `role="progressbar"`, `aria-valuenow={percentage}`, `aria-valuemin={0}`, `aria-valuemax={100}`, `aria-label` describing what is being measured
    - TogglePills: container has `role="radiogroup"`, each pill has `role="radio"` + `aria-checked={true|false}`
    - Numpad: all keys are `<button type="button">`, digits have text content as accessible name, backspace has `aria-label="Borrar"`
    - FloatingInput: real `<label htmlFor>` element (not just visual), `aria-invalid="true"` on error state, `aria-describedby` pointing to error message element
    - Modal: `role="dialog"`, `aria-modal="true"`, `aria-labelledby` pointing to title element
    - StatusDot: decorative -- should have `aria-hidden="true"`
    - MoneyAmount: `aria-label` with full formatted amount for screen readers

    **4. Semantic HTML (UX_RULES.md 9.2):**
    - `<nav>` for sidebar and bottom tab bar
    - `<main>` for primary content area (in layout.tsx)
    - `<section>` with `aria-labelledby` for dashboard sections
    - Tables use `<table>`, `<thead>`, `<tbody>`, `<th scope="col">`
    - Forms use `<label>` associated with inputs via `htmlFor`
    - Decorative icons: `aria-hidden="true"`
    - Functional icons: `aria-label` with description

    **5. Reduced Motion (STYLE_GUIDE.md > Identidad Visual):**
    - Verify `@media (prefers-reduced-motion: reduce)` in globals.css disables:
      - `animate-status-pulse` (StatusDot)
      - `animate-scanline-reveal` (pixel-dissolve)
    - Verify the `clip-path: none` override is present for scanline animation
    - Check any inline animations or transitions are also guarded
    - Button press `scale(0.98)` interaction can remain (subtle, not motion-sensitive)

    **6. Keyboard Navigation:**
    - All interactive elements reachable by Tab
    - Enter/Space activate buttons
    - Escape closes modals and dropdowns
    - Arrow keys navigate within TogglePills radiogroup
    - Numpad keys all keyboard-accessible
    - No keyboard trap in any component

    **7. Screen Reader Support (UX_RULES.md 9.3):**
    - Toasts use `role="status"` and `aria-live="polite"` (check sonner config in layout.tsx)
    - BatteryBar has screen-reader-friendly label describing the metric and percentage
    - Monetary amounts have parseable `aria-label` (e.g., "$1,500.75")

    For each violation found: fix the code directly. Document what was wrong and what was changed. Prioritize: contrast failures > missing ARIA > focus ring gaps > semantic HTML > reduced motion.
